# Users Directory

A small React Native (Expo) app that browses people from the
[DummyJSON Users API](https://dummyjson.com/docs/users). It demonstrates a
paginated, searchable, sortable list with sticky alphabetical section headers, a
detail screen with a scroll-driven collapsible header (Reanimated), a small
reusable design system, and unit/integration + E2E tests.

| | |
|---|---|
| **Platforms** | iOS + Android (single codebase, no platform-only paths) |
| **Language** | TypeScript (strict) |
| **Navigation** | React Navigation (native stack) |
| **Data** | TanStack Query (React Query) |
| **Animation** | react-native-reanimated |
| **Tests** | Jest + React Native Testing Library + MSW; Detox (E2E) |

## Features

- **List** — infinite pagination (`FlatList`/`SectionList`, virtualized), pull-to-refresh, loading / error+retry / empty states.
- **Search** — server-side, debounced, with a clear/reset control.
- **Sort** — server-side alphabetical by last name, A–Z / Z–A toggle.
- **Sticky section headers** — grouped by last-name initial, sticky until the next letter.
- **Detail** — fetch by stable id with instant cache-seeded render, plus a scroll-driven collapsible header.
- **Design system** — `Text`, `Avatar`, `Input`, `Button`, `Card` primitives with accessible defaults.

## Getting started

> This is an Expo **dev-client** app (it uses `expo-dev-client`, Reanimated, and
> other native modules). **Expo Go will not work** — you build and run a real
> dev build with `expo run:*`.

### Prerequisites

- **Node** ≥ 18, npm.
- **iOS:** macOS + Xcode + Command Line Tools, **CocoaPods** (`brew install cocoapods`).
- **Android:** Android Studio + SDK, an emulator (AVD) or device, `ANDROID_HOME` set.
- Network access to `https://dummyjson.com` (the app fetches live data).

### Install

```bash
npm install
```

### Run on iOS

```bash
npm run ios          # = expo run:ios — generates native project, pods, builds, launches
# pick a simulator:  npx expo run:ios --device "iPhone 16 Pro"
# physical device:   npx expo run:ios --device
```

### Run on Android

```bash
# Have an emulator running (or a device connected with USB debugging):
emulator -list-avds && emulator -avd <name> &   # or use Android Studio
adb devices                                      # confirm it's attached

npm run android      # = expo run:android — generates native project, builds, installs, launches
```

The first native build is slow (it compiles the dev client). After it's
installed, JS changes don't need a rebuild — run `npx expo start --dev-client`
and press `i` / `a`, or `r` to reload.

> The native `ios/` and `android/` folders are **gitignored** and regenerated by
> `expo prebuild` / `expo run:*` from `app.config.ts` — they are not committed.

## Testing

### Unit / integration (Jest + RNTL)

```bash
npm run typecheck     # tsc --noEmit
npm run lint          # eslint
npm test              # jest (jest-expo preset)
npm run test:watch
```

26 tests across 6 suites. Coverage highlights:

- **Design system** — `Button` (label, press, disabled/loading blocks press, a11y role/state).
- **Home screen** — loading→data, error+retry, empty, search filter + clear-reset, pagination request, sort reordering + sort request params, sticky section headers + ordering.
- **Detail screen** — instant placeholder render from cache, then detail-only fields after the by-id fetch.
- **Pure logic** — `buildSections` grouping (`sections.test.ts`), the MSW boundary smoke test.

**Network mocking — MSW at the network boundary.** Tests run the real
`apiClient` + React Query against [MSW](https://mswjs.io) handlers in
`src/test/server.ts` (started/stopped centrally in `jest.setup.ts`), so the
whole fetch → parse → cache → render path is exercised, not stubbed. Fixtures
live in `src/test/fixtures`; per-test error/empty/sort overrides use
`server.use(...)`. Each test gets a fresh `QueryClient` (`retry: false`,
`gcTime: 0`) via `src/test/render.tsx`.

A few setup notes worth knowing (the New Architecture made these necessary):

- **Reanimated 4 mock** — the package's own `/mock` initializes
  `react-native-worklets` and crashes under Jest, so a minimal hand-rolled mock
  lives at `src/test/mocks/reanimated.js`. Types still come from the real
  package during `tsc`.
- **MSW v2 + jest-expo** — needs `.mjs` transform, broadened
  `transformIgnorePatterns`, and `moduleNameMapper` pointing `msw`/`msw/node` at
  their CJS builds (MSW sets the `react-native` export condition to `null`). All
  in `jest.config.js` / `jest.setup.ts`.
- **Debounced-search test uses real timers** — under RNTL v14 the async
  `fireEvent` doesn't compose with fake timers for a debounced `setState`; the
  350 ms debounce resolves well inside the `findBy*` poll window, so the test is
  deterministic.

### End-to-end (Detox)

The Detox suite exercises the full required path against a real native build:

1. Launch → Home loads the first page, sorted by last name A→Z (Gabriel Adams, id 31, leads).
2. Search "Emily" → list filters to the Emilys; Gabriel Adams (id 31) is filtered out.
3. Tap clear → the full sorted list returns (Gabriel Adams is the first row again).
4. Re-search and tap a user → the detail screen (with its collapsible header) opens.

A second spec, `e2e/sort.test.ts`, exercises the A–Z ↔ Z–A sort toggle by
asserting the top row flips between Gabriel Adams (id 31, ascending) and Layla
Young (id 191, descending).

Specs: `e2e/starter.test.ts`, `e2e/sort.test.ts`; config: `.detoxrc.js`; runner:
`e2e/jest.config.js`.

```bash
# All-in-one (prebuild -> build -> test), fail-fast:
npm run e2e:ios           # iOS only
npm run e2e:android       # Android only
npm run e2e:all           # both, sequentially (macOS)

# Or run each step manually:
# iOS
npm run prebuild:ios
npm run e2e:build         # detox build -c ios.sim.release
npm run e2e:test          # detox test  -c ios.sim.release

# Android
npm run prebuild:android
npm run e2e:build:android # detox build -c android.emu.release
npm run e2e:test:android  # detox test  -c android.emu.release
```

**Why release builds for the default E2E?** The default app build is an Expo
**dev client** that loads JS from Metro over the network; under Detox's cold
launch it just shows the "enter a URL" launcher and the test hangs. The `e2e:*`
scripts therefore use **release** configs, which embed the JS bundle — the app
is self-contained, needs no Metro, and boots straight into Home (deterministic,
as Detox recommends). Expo signs the release variant with the debug keystore, so
no signing setup is required.

**Running the debug (dev-client) E2E.** Debug builds are useful for fast
iteration and breakpoints. Two things make them work where a naive run hangs on
the launcher:

1. **Metro must be running**, and
2. **the app must be deep-linked at it** — otherwise the dev client sits on the
   launcher screen.

Both are handled for you:

```bash
npm run e2e:ios:debug       # prebuild -> debug build -> (boot Metro) -> test -> stop Metro
npm run e2e:android:debug
```

`e2e/with-metro.js` boots `expo start --dev-client`, waits until it's serving,
runs the suite, then shuts it down. The launch itself goes through
`launchAtHome()` (`e2e/launch.ts`): when `E2E_DEV_CLIENT=1` (set by the debug
test scripts) it deep-links the dev client at the
`usersdirectory://expo-development-client/?url=http://<host>:8081` URL (`<host>`
is `localhost` on iOS, `10.0.2.2` on the Android emulator) so it skips the
launcher and loads our bundle. **Delivery differs per platform:**

- **iOS** — cold-launch, then `openURL`. Passing `url` to a cold `launchApp` is
  dropped by some iOS dev-client versions (you stay on the launcher).
- **Android** — deliver the URL as the cold-launch intent (`launchApp({ url })`).
  `openURL` only sends a VIEW intent to the already-foreground launcher, which
  doesn't reload it (the menu just stays up).

The first run then waits on Metro's **initial bundle build**, so the first spec
query allows 60s. To run against an already-running Metro, use the lower-level
`npm run e2e:build:debug && npm run e2e:test:debug` (append `:android` for
Android).

The dev client otherwise shows a one-time **"developer info" onboarding bottom
sheet** (and auto-opens the dev menu) on a fresh install, which blocks the test
behind a popup. A small config plugin in `app.config.ts` (`withQuietDevMenu`)
disables this at the native level — it writes `EXDevMenuShowsAtLaunch=false`,
`EXDevMenuIsOnboardingFinished=true`, and `EXDevMenuShowFloatingActionButton=false`
into the iOS `Info.plist` and the Android `<application>` meta-data at prebuild,
which expo-dev-menu reads on launch. The dev menu is still reachable via shake /
⌘D; only the launch-time popups are suppressed. (Baked at build time, so it takes
effect on the next `prebuild` + build.)

**E2E prerequisites:** macOS + Xcode (iOS) / Android Studio with any AVD
(Android), CocoaPods, **`applesimutils`** (`brew tap wix/brew && brew install
applesimutils`), `npm install`, and network access. The header-collapse animation
is not asserted in E2E (Detox runs with animations disabled, so the crossfade
never plays under test); it's covered by the unit/integration suite instead.

`.detoxrc.js` **auto-detects** a device so no per-machine edit is needed: it
picks the newest available iPhone simulator (`xcrun simctl list`) and the first
installed AVD (`emulator -list-avds`). To pin a specific device — or in CI — set
`DETOX_IOS_DEVICE` / `DETOX_AVD_NAME` (e.g. `DETOX_AVD_NAME=Pixel_7_API_34 npm
run e2e:test:android`).

**E2E hits the real DummyJSON API** (not MSW). DummyJSON is stable and
deterministic, so the spec asserts on fixed records under the app's default
last-name-ascending sort: the first row is Gabriel Adams (id 31), and
`?q=Emily` returns Emily Brown (id 103) + Emily Johnson (id 1) and excludes
Gabriel Adams. Detox uses auto-sync + `waitFor(...).withTimeout(...)` for the
Home/search steps, then **disables synchronization** before opening the detail
screen: Reanimated's UI-thread runtime keeps Detox's idle check from ever
settling on the animated header, so auto-sync would otherwise hang there.

### End-to-end (Maestro)

The journeys are also written for **[Maestro](https://maestro.dev)** in
`e2e/maestro/users-directory.yaml` — an alternative runner that drives the UI
from YAML and matches elements by the same `testID`s. The Detox `starter` + `sort`
specs are combined here into a **single flow** (load → sort toggle → search →
clear → open detail) over one app launch. Maestro polls the screen, so the Detox
sync/animation workarounds aren't needed; it reads as a plain list of taps and
assertions.

**Install the Maestro CLI** (once). It needs a **JDK (11+)** on your PATH first
(`brew install openjdk`), then:

```bash
curl -fsSL https://get.maestro.mobile.dev | bash   # installs to ~/.maestro
export PATH="$PATH":"$HOME/.maestro/bin"            # add to your shell profile
maestro --version                                   # verify it's on the PATH
```

(`applesimutils` — already needed for Detox — also lets Maestro manage iOS
simulators.) Then build + install the app and run the flow:

```bash
# Build + install the app on a booted simulator/emulator (e.g. one of):
npm run prebuild:ios && npm run e2e:build        # or: npx expo run:ios / run:android

npm run maestro:test                             # whatever single device Maestro auto-detects
npm run maestro:test:ios                          # the booted iOS simulator
npm run maestro:test:android                       # the running Android emulator
npm run maestro:test:all                            # iOS then Android, sequentially
# or directly: maestro test e2e/maestro/users-directory.yaml
```

Maestro drives **one device per run** and auto-picks an available one (so with
only an Android emulator up, `maestro:test` runs Android). The `:ios` / `:android`
scripts select the booted simulator / emulator explicitly via `--device`, and
`:all` runs both in turn.

Both flows start with `launchApp` and assume a build that **boots straight to
Home** — i.e. a **release** build (recommended; the bundle is embedded), or a
debug dev-client build with Metro running. For the dev-client case, start Metro
(`npx expo start --dev-client`) and add an `openLink` step after `launchApp` to
deep-link the client at it:

```yaml
# iOS simulator:
- openLink: "usersdirectory://expo-development-client/?url=http%3A%2F%2Flocalhost%3A8081"
# Android emulator (host loopback is 10.0.2.2):
- openLink: "usersdirectory://expo-development-client/?url=http%3A%2F%2F10.0.2.2%3A8081"
```

Like Detox, Maestro hits the real DummyJSON API and asserts the same fixed
records (Gabriel Adams id 31 first ascending, Layla Young id 191 first
descending, `Emily` search → Emily Johnson id 1). The header-collapse animation
isn't asserted (same rationale as Detox).

> **Status:** the offline-verifiable checks (`typecheck`, `typecheck:e2e`,
> `lint`, `npm test`, Detox/Jest config loading, `expo prebuild`) were run and
> pass. The native `detox build`/`test` run requires a local macOS + simulator
> toolchain with `applesimutils`; the steps above are documented as
> reproducible.

## Architecture

Three layers with a strict, one-directional dependency rule (enforced by ESLint
`import/no-restricted-paths`):

```
ui/        generic design system — knows nothing about users or the API
features/  domain logic + screens — composes ui/ and lib/
lib/       cross-cutting infra — apiClient, queryClient, env
app/       wiring — navigation container + providers
```

```
src/
  app/                 NavigationContainer, native-stack navigator, typed routes
  features/users/
    api/               endpoints + typed query-key factory + hooks (useUsers/useUserSearch/useUser)
    components/        UserListItem, SearchBar, SortToggle, SectionHeader, UserDetailHeader,
                       UserDetailContent, UserDirectoryList, CollapsibleHeaderScrollView
    lib/               buildSections (alphabetical grouping), flattenUsers (page-shape seam)
    screens/           HomeScreen, UserDetailScreen
    types/             User domain types, UserSort
  ui/
    components/        Text, Avatar, Input, Button, Card, QueryStateView
    theme/             static design tokens (light-only)
  lib/                 apiClient, queryClient, env
  test/                renderWithProviders, MSW server, fixtures, reanimated mock
e2e/                   Detox spec + config
docs/adr/              architectural decision records
```

**State ownership** is deliberate:

- **Server state → React Query.** Lists, search, and detail are cached, deduped, and invalidated by React Query. Server data is never copied into `useState`.
- **Navigation state → route params.** The detail screen receives a `userId` (the stable id), never a full user object.
- **Local UI state → `useState`.** Search text (inside `SearchBar`), sort order, scroll offset.

No global client-state library — it would be over-engineering at this scope. The
conventions are documented in `.claude/rules/*.md`, the domain glossary in
`CONTEXT.md`, and decisions in `docs/adr/`.

## Key decisions & tradeoffs

### Screen composition (deep modules)

Screens stay thin: they own local UI state (search term, sort order) and
navigation, and compose modules that each hide one concern behind a small
interface.

- **`QueryStateView`** (`ui/`) owns the loading / error / retry / empty
  presentation. Both screens were re-implementing the same branch; the four
  states now live and are tested in one place, and each screen renders only its
  ready content.
- **`UserDirectoryList`** (`features/users/components`) owns the virtualized
  `SectionList` — perf tuning, the Android/Fabric `removeClippedSubviews`
  workaround, the footer spinner, and the pagination gating — so `HomeScreen`
  passes ready-made sections and the directory's status, not list config.
- **`CollapsibleHeaderScrollView`** owns the scroll-offset `SharedValue`, the
  animated scroll handler, and the content inset that clears the header.
  Reanimated no longer leaks into `UserDetailScreen`; callers pass header data
  and body content.

The result: `HomeScreen` and `UserDetailScreen` read as composition, and the
extracted modules are unit-testable through their own interface rather than only
through a full screen render.

### Data fetching

- **TanStack Query** for all server state. List and search use `useInfiniteQuery` with `PAGE_SIZE = 30` (the API default; balances payload vs. round-trips); the next `skip` is computed from `skip + limit` and stops at `total`.
- A single **`apiClient`** (`lib/apiClient.ts`) centralizes the base URL (from `app.config.ts` → `expo-constants` → `lib/env.ts`), a 10 s `AbortController` timeout merged with React Query's `signal` (so requests cancel on unmount/param change), an `res.ok` check, a normalized `ApiError { status, message }`, and typed JSON. Timeout-vs-cancellation is detected with a closure flag + the caller signal's `aborted` flag rather than `DOMException`, which isn't reliable under Hermes.
- **Detail** uses `useQuery(['user', id])` seeded with `placeholderData` scanned from the list/search caches, so navigating in renders **instantly**, then confirms against `/users/{id}`. DummyJSON returns the full user in list responses too, so this is also the textbook "fetch by stable id" the brief asks for and stays correct on a cold cache / deep link.
- `staleTime: 60s` so returning to Home doesn't refetch; `keepPreviousData` on search/sort to avoid flicker and full-screen spinner flashes.

### Search

Server-side via `GET /users/search?q=`, **debounced ~350 ms** (so we don't fire a
request per keystroke), with `keepPreviousData` to avoid flicker and a
clear/reset control. The raw input text lives inside `SearchBar` so keystrokes
don't re-render the list before the debounce fires; only the debounced term is
lifted. An empty query shows the normal paginated list (search isn't called with
an empty term). **Server-side is the default over a naive client-side filter**
because the dataset is paginated — filtering only the loaded rows would silently
miss everyone else, giving results that depend on how far you'd scrolled.

**Hybrid path** (see [ADR 0002](docs/adr/0002-hybrid-search.md)). On top of that
default, search switches to instant local filtering *only when it's provably
correct* — when the list cache is complete:

- **Cache complete** (list fully paged in for the active sort → the whole
  directory is in memory): filter locally on `firstName`/`lastName`/`email` and
  **disable the search query** — zero requests, works offline, results exhaustive.
- **Cache incomplete + online**: hit the server (a partial cache can't tell "3
  matches" from "3 of 20").
- **Cache incomplete + request fails (offline)**: fall back to best-effort local
  matches and show an *"Offline — results may be incomplete"* notice
  (`offline-notice`); never silently incomplete.

All of this is encapsulated in `useUserDirectory` (the screen just reads
`isPartialResults`). It **fails safe**: request-cutting only engages once the
cache completes — realistic for the ~208-user set (7 pages), and on a genuinely
huge dataset it simply no-ops back to server search rather than breaking
correctness. Local field parity with the server is approximate (mirrors the
fields DummyJSON is confirmed to index and we model).

### Sort

Server-side via `sortBy=lastName&order=asc|desc`, toggled A–Z / Z–A (default
A–Z), applied to **both** list and search. **Server-side is required, not a
preference:** with infinite pagination a client `.sort()` only reorders the rows
already fetched, so the next page would break global ordering. The sort is part
of the React Query key, so each variant is its own cache entry (toggling back is
instant from cache); `keepPreviousData` keeps the current order on screen while
the new order loads.

### Sticky section headers

The list is a `SectionList` grouping users by **last-name initial** — which
matches the last-name sort, so sections stay contiguous and correctly ordered in
both directions (`buildSections` is a single O(n) pass over the
already-server-sorted data; non-alpha last names fall into a `#` bucket, which
assumes the clean DummyJSON data). Headers are sticky in both list and search.
Switching from `FlatList` meant dropping `getItemLayout` (a correct SectionList
implementation is error-prone and ships no first-party helper) — fixed row +
header heights keep measurement cheap; revisiting this is in "what I'd improve".

> Note: the index letter follows the **last name** (the sort key), while a row's
> prominent text is the first name — intentional, but a screen reader announces
> the section as the last-name initial.

### Animation (Reanimated)

The detail screen has a **scroll-driven collapsible header**. An
`useAnimatedScrollHandler` feeds the scroll offset to `useAnimatedStyle`
worklets on the UI thread: the header height shrinks to a compact bar, the large
centered/bold avatar+name block fades and drifts up, and a compact title
crossfades in — left-aligned, normal-weight, sliding leftward into place. The
header blends into the screen background (no separator). It honors the OS
**reduce-motion** setting (stays expanded). E2E observes the compact title
becoming visible after scrolling.

### Scalability / performance

- Virtualized list with a stable `keyExtractor` (the user `id`, never index), `React.memo` rows, and `useCallback`-stable `renderItem`/`renderSectionHeader`/`keyExtractor`/handlers. No inline styles in render paths.
- Tuned `initialNumToRender` / `maxToRenderPerBatch` / `windowSize`; fixed `ROW_HEIGHT` / `SECTION_HEADER_HEIGHT`.
- **`removeClippedSubviews` is intentionally OFF.** On the New Architecture (Fabric) it crashes Android (`ReactClippingViewManager` `addViewAt … IndexOutOfBounds`) when list contents churn — e.g. clearing the search term. Virtualization already bounds mounted rows, so the offscreen-memory win wasn't worth a crash.
- `expo-image` for avatars (memory/disk cache, downsampling) so scrolling never blocks on the network; debounced search; `placeholderData`/`staleTime` to minimize fetches; in-flight cancellation via `AbortController`.

### Accessibility

Roles/labels/states on interactive elements, 44 pt minimum hit targets, list
rows composed into a single screen-reader node, and a canonical `testID`
contract shared by RNTL and Detox (see `.claude/rules/accessibility.md`). The
sort toggle models direction with `accessibilityValue` (not a boolean
`accessibilityState`, which would misrepresent it).

### Error handling

`apiClient` throws a normalized `ApiError`; React Query surfaces
`isError`/`error`; screens render a readable error message with a labeled
**Retry** that calls `refetch()`. Loading and empty states are explicit and
screen-reader readable. Errors are never swallowed silently.

### Platform & build

- **Expo managed workflow with a dev client** (ADR-0001) — Detox needs a real native build, so Expo Go isn't sufficient. Native folders are generated by `expo prebuild` and gitignored.
- **Light-only theming** — a static frozen tokens module imported directly (no `ThemeProvider`/context), since dark mode is out of scope. Tokens are structured so a provider could be added later.
- **Android cleartext** — the dev/Detox config (`@config-plugins/detox` with `subdomains: '*'`) emits a network-security config permitting cleartext to any host, so the Metro dev server is reachable on any LAN IP. This applies to all build types; a production app would scope it to debug only.
- `@config-plugins/detox` (v11) peers to `expo@^53`; it's build-time only, so its `expo` peer is pinned to 56 via a `package.json` `overrides` entry (reproducible on a clean `npm install`).

## What I'd improve with more time

- **Run Detox in CI** against a managed simulator/emulator (the suite is written and config-verified; only the native run is environment-gated here).
- **`getItemLayout` for `SectionList`** (or migrate to `@shopify/flash-list`) if profiling shows scroll jank on very large datasets.
- **Dark mode** via the existing token system + a thin `ThemeProvider`.
- **Richer sort** — sort by first name too, persist the choice, and add an alphabet scrubber / jump-to-letter index.
- **Scope Android cleartext to debug builds only** (currently all build types) and drop the `@config-plugins/detox` `overrides` once an SDK-56 build ships.
- **Resilience UX** — retry/backoff and an offline indicator; an image error fallback state on `Avatar`.
- **Test polish** — add a search+sort integration test and silence the benign React Query `act()` warning so the suite can fail on console errors.
